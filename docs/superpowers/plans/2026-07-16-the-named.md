# The Named Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stop the almanac's Gods section deciding a pantheon's lead by block index, so every pantheon in a multi-people world names the people who hold it (hornvale#1).

**Architecture:** Attribution becomes a property a pantheon carries, not a place it sits. `PantheonBlock` gains `attribution: Option<PantheonAttribution>` replacing three empty-string sentinels; `render` drops its `i == 0` special case and reads that field; the worldgen builder — the composition root — owns the policy and withholds attribution when the world has fewer than two placed peoples, matching the People section's existing rule via one shared predicate.

**Tech Stack:** Rust 2024, workspace crates `windows/almanac` + `windows/worldgen`, `cargo nextest`, mdBook.

**Spec:** `docs/superpowers/specs/2026-07-16-the-named-design.md` (approved at G3, 2026-07-16).

```rust
/// Every peopled species holding a flagship settlement, in registry
/// (alphabetical) order.
///
/// The single basis for the almanac's name-the-species-only-when-there-is-
/// more-than-one convention. Both the People section (`settlement_lines`)
/// and the Gods section's pantheon attribution read this one predicate, so
/// the two sections cannot drift into naming a world's peoples in one place
/// and withholding them in the other.
/// type-audit: bare-ok(identifier-text: species leg of return)
pub fn placed_peoples(world: &World) -> Vec<(&'static str, hornvale_settlement::VillageInfo)> {
    hornvale_species::registry()
        .keys()
        .filter_map(|name| flagship_of(world, name.0).map(|v| (name.0, v)))
        .collect()
}
```
