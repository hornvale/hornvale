# Deep Time — retrospective

**Merged:** 2026-07-09

**The null control caught two real design defects before merge, not after.**
The zero-forcing pin was preregistered as a structural proof, not a vibe
check — flat forcing must produce an exactly-empty glacial record, by
construction, for every seed. Holding the model to that bar caught two
defects a looser review would have let through. First, an early caloric-
summer index was anchored to a fixed Earth-like reference obliquity; a
world drawn with a different baseline tilt then read as permanently forced
even under a zero-forcing pin, because "flat" was being measured against
the wrong zero. The fix made the index an anomaly against the world's *own*
mean obliquity, so the dead band lands at the world's own baseline
regardless of what it drew. Second, an early version of the era loop let
the ice-extent envelope and the era's own `ice_fraction` summary derive
from two separately-computed masks; under most forcing they agreed, but
nothing forced them to, and the null control's "zero advance everywhere"
claim depended on both reading the identical thing. The fix routed both
through one mask, computed once. Neither defect would have failed a
determinism test (both were internally consistent and reproducible, just
wrong) — only a preregistered *physical* invariant, checked exactly, forced
them into the open. The lesson generalizes past this campaign: a
determinism gate proves a wrong answer is stably wrong; a preregistered
null control is what proves the answer is right.

**Reading the engine before finalizing a plan changed the design, again.**
The plan's first pass at the era loop assumed the existing mean-temperature
path read a world's obliquity per era, the same way the full climate
generator's seasonal term does. It does not — `mean_temperature` is a
latitude/insolation baseline plus a lapse term, obliquity-blind by
construction, only the seasonal-swing and moisture terms read it. Finding
that by reading `domains/climate/src/temperature.rs` directly, rather than
inferring it from the function's name, meant the per-era diagnostic could
be built as a cheap direct call into that same function (bit-identical to
what a full climate rebuild would produce) instead of assuming a full
climate regeneration was required at every era. This is the same rule
named in more than one prior campaign's retrospective — verify a
mechanism against the code that computes it before a plan leans on an
assumption about that mechanism — and it shaped this campaign's central
performance decision below, not just a footnote.

**A late fidelity-vs-performance call was cheap because it changed no
observable behavior.** The first working era loop paid for a full climate
rebuild — moisture field, biome classification, habitability map — on each
of the ~25 coarse eras, to read back only the one field (mean temperature)
the ice diagnostic actually needed. Once the obliquity-blindness above was
confirmed, the loop was restructured to call the cheap temperature-only
path for 23 of those eras and reserve the expensive full rebuild for the
single era that turns out to be the glacial maximum (the only one whose
habitability mask, for refugia, is ever read). Twenty-six climate builds
became two. Because the cheap path reproduces the expensive one's
temperature field exactly, this was a pure performance change — every
committed artifact regenerated byte-identical — which is what let it land
late, after the physics was already settled, with no re-verification of
the science it sits underneath.

**The newtype decision paid for itself inside this campaign, not just
across future ones.** `Celsius` and `TempAnomaly` were split into distinct
types specifically because an earlier iteration of the ice-diagnostic
function *did* mix up an absolute reading and a difference-from-present at
a call site, silently, and it cost a real debugging pass to find. Sealing
`TempAnomaly`'s constructor so the only production paths are `Celsius`
subtraction and one crate-internal offset function closed that mistake at
the type level rather than trusting the next call site to remember the
convention. Worth naming: the type was added mid-campaign, after the bug
it fixed had already been caught by hand once — a case where the
newtype-per-quantity discipline was reached for reactively and still
turned out to be the right call, not merely a rule applied by rote at
crate-scaffold time.

**No determinism friction in the forward-run history.** The concern going
in was that a genuine million-year forward march, rather than a lookup,
would introduce some new source of non-reproducibility (float accumulation
order, iteration order over the mesh, threading). None materialized: the
integrator and era loop are byte-identical across repeated runs from the
same seed and pin, verified directly in the property battery and again at
the composition-root level, with no special-casing required. The
determinism discipline the kernel already enforces (no wall-clock, no
hash-map iteration, `total_cmp` float ordering) generalized to a forward
integration without needing new rules.
